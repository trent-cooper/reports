Overtime Calculation & Allocation

How two WFM systems calculate the same employee's overtime — and charge it to different locations

System 1

DW Labor Aggregates Exporter

A cron job running every 15 minutes, recalculating labor costs for each location throughout the day. Sends aggregated totals to the Data Warehouse for reporting dashboards. A DynamoDB stream monitors for punch changes and queues affected days/locations for reprocessing on the next cron run.

System 2

Payroll Preview (getPayroll)

Runs on-demand when a manager opens the payroll screen. Calculates each employee's hours broken down by every location and job code they worked at across the full pay period.

Both systems agree on total OT hours for an employee. The difference is which location gets "charged" that overtime cost. This presentation uses real payroll data to show exactly how and why.

If a manager edits a punch from Tuesday on Thursday afternoon, the DynamoDB stream detects the change and queues Tuesday + that location for reprocessing. The next cron run at :15 picks it up and recalculates Tuesday's numbers.

The DW dashboard is a near-real-time, rolling view of labor cost. It doesn't calculate once and freeze — it updates every 15 minutes as employees punch in/out, and retroactively when historical punches are edited.

Why Are They Different?

It comes down to one thing: the order hours are processed

DW Exporter: Chronological Order

Processes all punches in the order they actually happened. When it's calculating for a specific location, it sees other-location hours as "already happened" — they advance the OT counter, but only the current location's hours generate dollars.

The Sunday Paradox

Sunday's 10.08h at the home store is the starkest example. In chronological reality, Jamie was already 3.98 hours into overtime before Sunday started (43.98h through Friday). Every minute on Sunday is overtime.

But in the payroll preview, Sunday is queue position 4 (primary). Only 39.54h of primary-location hours have been counted. Sunday is still regular time. The overtime gets pushed to the secondary locations that come later in the queue.

DW Exporter says Sunday is:

10.08h OVERTIME

Payroll Preview says Sunday is:

10.08h REGULAR

When Does This Matter?

Practical implications for managers and reporting

Dashboard vs. Payroll Screen Mismatch

A manager at 910883 checking the DW dashboard sees $668.60 in labor cost for Jamie (including 10.08h OT). But when they open the payroll preview, they see $593.10 with zero overtime. Both are "correct" — they're answering different questions.

The Employee's Paycheck Is Unaffected

Jamie gets paid ~$916 either way. The allocation difference is purely about which location's P&L absorbs the overtime premium. This is a cost attribution question, not a compensation question.

Shared Employees Create Larger Gaps

The more an employee bounces between locations, the bigger the discrepancy. Jamie's case shows a $75.50 gap at the home store alone. An employee working at 5+ locations in a heavy-hours week could produce gaps of $200+.

Summary

DW Exporter

  • Cron every 15 minutes, live-updating
  • DynamoDB stream queues retroactive recalculations
  • Processes each location independently
  • Chronological punch ordering
  • Cross-location hours: counted but not costed
  • OT charged to the location where the employee is physically working when the threshold is crossed

Payroll Preview

  • On-demand, triggered by manager
  • Processes entire employee across all locations
  • Primary-location-first queue ordering
  • Cross-location hours: fully costed, but processed last
  • OT charged to secondary (borrowed) locations, protecting the home store's regular-hour budget
The total overtime hours and employee compensation are identical in both systems. The difference is strictly in location-level cost attribution: which store's labor budget absorbs the OT premium. The DW exporter answers "what did it cost to run this store?" while payroll preview answers "what does this employee's paycheck look like from this store's perspective?"
